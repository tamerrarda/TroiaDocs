This page draws a clear line between what Troia demonstrably does today and what it deliberately leaves for later. It is written to be read the same way by a newcomer and by a reviewer looking for the catch: everything is stated plainly, including the limits, because a hidden risk is far more damaging than a disclosed one.

Troia is a custodial settlement bridge on the Stellar network. A shopper in Turkey pays in lira with a Troy card through the payment processor iyzico; Troia settles the merchant in US-dollar value as USDC from a pre-funded pool it holds on Stellar; the small spread between the two currencies is the revenue. What follows is delivered as a proof-of-concept on Stellar's test network — a full working system exercised end to end, but with no real money at stake.

## The frame: prove the core first

Troia was built from the inside out. The parts that keep money safe, and the part that lets an outsider verify each payment for themselves, were built and hardened first — before any real rails were connected. Only then came the live payment and blockchain connections, and only then the storefront on top.

This is a deliberate sequencing choice rather than a way of avoiding the hard problems. The safety and verification layers are proven before real value can move through them, so the guarantees can be demonstrated rather than merely asserted.

The test network is where those guarantees are exercised from one end to the other with no real-money risk, because the USDC is minted by Troia itself for testing (explained below). The important point is that the core is not a throwaway prototype: the settlement logic, the defences against double payment, the solvency checks, the frozen price, and the verification tool are the foundation a production system would build on.

Building on it is still not a switch to flip. Going to production is a change of configuration and a few concrete implementations rather than a rewrite — but two named engineering gaps have to close first, both listed below and both explained on [Toward mainnet](./toward-mainnet.md).

## What is proven today

Everything in this section is covered by automated tests that must pass for every change, and the first several run offline, with no network connection and no secret credentials.

- **The settlement order that makes the system safe.** Sending USDC is irreversible; a lira card charge can be reversed the same day. Troia always performs the reversible charge first and the irreversible payout last, so a failure late in the flow unwinds by cancelling the same-day card sale rather than stranding money. Every step of a payment's life, and every permitted move between steps, is written down and tested.
- **Two independent defences against paying a merchant twice.** Each payout is pinned to a specific blockchain sequence number — a per-account counter the network accepts only once — so a duplicate attempt is rejected by the protocol itself. Behind that, the pool contract keeps its own record of which orders it has already paid and refuses to pay the same one again.
- **Solvency checked in two places.** Troia's backend sets aside pool funds before a shopper can be charged, and the pool contract independently confirms it holds enough before it releases anything. A request that cannot reserve funds is refused up front, before any charge is possible.
- **Pricing the client cannot influence.** The lira price is computed entirely on Troia's servers from four transparent parts: the live mid-market exchange rate, a commission sized to the risk of the roughly three-week settlement wait, a fixed margin, and the payment processor's fee passed through and grossed up so the net still covers the rest after the processor takes its cut. Any price or currency sent by the client is ignored; the shopper is charged exactly the frozen quote.
- **A deterministic exchange rate.** The rate is taken from the median of several distinct market sources, with a check that enough independent sources agree, a staleness limit, and a rule to fail closed when they disagree. No model or guess is involved, and the tested path uses supplied quotes rather than a live feed.
- **The self-verifying evidence tool.** For every payment, Troia keeps independent records of what was intended, what was signed and submitted, and what settled on-chain. An offline tool re-derives the outcome and confirms the three agree, with the network blocked. This is the reviewer's centrepiece, and it works today. See [Reconciliation](./reconciliation.md).
- **What Troia knows about money survives a crash.** Every fact the system relies on to move money is written to an append-only file before it is believed. Those files have a stated contract for what a crash may and may not do to them: a half-written record can only ever sit at the very end of a file, where it is repaired and reported, while a record that was fully written and later damaged is treated as fatal rather than quietly discarded. If the store ever refuses a write, the process stops instead of carrying on with an unrecorded payment.
- **The chain answers for itself.** Troia does not take its own word for what happened. A background watcher reads the token contract's own transfer records and treats any money leaving the pool that Troia never authorised as a rogue payout — it can say so with confidence because a payout's identity is written to disk before the transaction is ever broadcast, so a transfer that is missing from that record could not have been ours. A second watcher re-checks each settlement against the chain, finding it by the identifier the pool contract itself indexes rather than by the transaction hash Troia recorded, and refuses to call an order reconciled unless the pool's code was never replaced, the amount announced matches the amount the token contract actually moved, and the transaction is still there. Where it cannot see, it says so rather than accusing anyone.
- **The pool contract on Stellar.** The Soroban smart contract that holds the custodial USDC pays out in a single atomic check-and-transfer, so its balance check and the transfer cannot fall out of step. It carries a replay guard, a pause control, and role-restricted administration, and its unit, integration, and conservation tests all pass.
- **The browser extension.** The demonstration storefront's checkout is recognised by a thin browser extension that offers to pay with a Troy card. It holds no keys, signs nothing, and fails closed rather than guessing. It has been proven live, end to end, into a real on-chain payout.

## What the test network deliberately does not prove

The following are conscious boundaries of a test-network proof-of-concept, documented here so they are never mistaken for claims.

- **The USDC is self-minted.** The pool is funded with Troia's own test-network USDC, which it can mint without limit. This fully exercises the solvency *mechanism* — the reservation and the contract's balance guard — but it does not prove *economic* solvency, meaning actually having bought the USDC in the first place. The automatic top-up runs today against a simulated purchase; only the real exchange trade that genuinely acquires inventory is deferred. This is the next funded milestone rather than a hole in the design: the decision to rebalance and the execution of it already sit behind separate, exercised interfaces, and production replaces one implementation behind them with a real exchange purchase and withdrawal. The mechanism is proven; what money buys is the inventory.
- **Signed is not the same as settled.** Troia proves what it signed cryptographically, in a way that survives restarts, and separately proves what settled — but only for as long as the chain still remembers it. A wiped test network or a transaction that never landed is reported honestly as unsettled, never as a false match.
- **The one irreversible loss window is named, not hidden.** In the narrow case where the USDC has already been sent but the lira charge cannot be reversed, the order is placed in a visible review state flagged with its evidence, never silently absorbed. When such a loss happens it is Troia's to absorb, never the customer's. On the test network no real value is at stake; this path exists to show the worst case is handled responsibly. It is not the only way a customer can be disappointed, though — see the stranded charge described below.
- **The payment processor runs in sandbox mode.** The lira leg runs against iyzico's sandbox with Troy test cards, and real charges have been carried out successfully. What the sandbox cannot reveal is the true settlement hold — the days iyzico blocks the funds before releasing them, which in practice is far longer than the advertised next-day figure. Because that delay cannot be measured in the sandbox, the commission uses a conservative researched estimate of about twenty-one days. The way Troia reads iyzico's success and decline responses, however, is calibrated against a real charge and iyzico's published response codes.
- **The unit economics are disclosed, not assumed.** The pricing model is complete and does not lose money by construction. The resulting all-in markup depends heavily on which card rail is used — roughly 7.8% on iyzico credit at current calm volatility, against roughly 3.7% on a bank virtual-POS debit rail — and the settlement-risk portion is data-driven, so it rises when the market grows volatile. Three levers close that gap, and all three are ordinary commercial work rather than research: the cheaper debit rail, a negotiated processor rate, and a shorter settlement hold, which shrinks the risk buffer faster than it shrinks the wait. The number is surfaced here because a markup hidden inside an exchange rate is precisely what this project exists to end.

## Proven live, and the limits that remain

Troia has now been driven end to end on the test network twice: a real Troy sandbox card charge automatically driving a real on-chain payout, with nothing hand-run in between. The second run also exercised the crash-durable records, both chain watchers, and a deliberate kill and restart of the server against the same stored state. The books matched the chain exactly, nothing was paid or recorded twice, and Troia's own payout was still recognised as authorised after the restart had erased everything held in memory. The evidence is on the [Deployments](./deployments.md) page.

What remains is genuinely operational, and it is listed rather than summarised away.

- **Both live runs were single manual smokes, not load or endurance tests.** Behaviour under many simultaneous checkouts is proven offline, including a check that removing the pool's lock makes the same workload over-commit, but it has never been driven against the live rails. It is worth being precise about the shape of that gap: the reservation performs its check and its commitment inside a single acquisition of the lock, no network call happens inside that lock at all, and the offline test deliberately injects a pause at the one point where the runtime could interleave anything. A live burst of traffic would therefore exercise a *narrower* window than the offline test already does. What remains untested is throughput rather than correctness.
- **Solvency assumes exactly one backend process.** The reservation gate is a lock held inside a single running process. Two copies of the backend against the same pool would each hold their own lock and could both reserve the last coin. The contract's own balance check is the second, independent shield, so the chain still cannot be overdrawn — but the backend half of the guarantee disappears. This matters on the day the demonstration is deployed to a platform that runs two instances, or overlaps them during a redeploy.
- **A crash between the charge and the payout strands the order.** Orders themselves are held in memory. If the process dies after the shopper has paid on the hosted form but before the payout begins, nothing durable records the charge, so on restart the order is neither driven forward nor cancelled and the merchant is never paid. No double payment is possible in that window, but the shopper has been charged and not settled, with no automatic unwind. On the test network the cards are valueless and this costs nothing; with real money it would be a genuine exposure. Durable order records close it, and together with the single-process lock above this is why a durable store is a production prerequisite rather than a nicety.
- **The pool's automatic refill can mint twice across a crash.** The guard against refilling the same order twice is durable, but it is read *before* the top-up moves and written only *after* it lands. A crash inside that window leaves a top-up that happened and was never booked; on restart the order is armed again and refills a second time. On the test network the error runs in the harmless direction — the pool ends up holding more than the books say, never less, so no customer is short-changed and no merchant unpaid — but the books and the chain then disagree permanently, and the drift alarm has something it can never clear. It matters far more later: in production that same step is a real purchase of USDC with real lira, so switching to a real exchange does not retire this bug, it is what makes it expensive. The fix is the discipline the payout path already follows — write the intent down before the money moves — and it must land before or with that switch, never after.
- **The payment-intent endpoint is unauthenticated.** Anyone who can reach the backend can open an order. That is not a theft path: the caller pays the lira themselves, the price is computed server-side, and every field is re-validated before anything moves — so a forged call can neither misroute funds nor dictate a price. What it can do is cost, because each accepted order reserves pool funds. A per-address cap is in place, which stops a naive flood but not a distributed one; a determined caller could still fill the pool with reservations and make honest shoppers see a refusal. On the test network the exposure is availability, never loss, and it is named here because it is worth closing before this runs anywhere public.
- **Two of the sharpest detectors have since fired on chain.** On 14 July 2026 the rogue-payout alarm caught a real unauthorised outflow — a payout sent straight to the contract by hand, bypassing the backend — and the diagnostic read that classifies a reverted payout was confirmed against a genuinely reverted transaction. What remains exercised by tests alone is the divergence verdict (a *different* transaction settled this order), the watcher's two "we could not see" states, a successful contract upgrade — only the permission gate is tested, because a real second version to upgrade into is future work and must never be rehearsed on the live pool, where a bad one would brick it — and load or endurance testing. Each of these is a testing-maturity gap rather than a defect: the code is there and passing.

The stranded charge and the single-process lock are closed by the same change — durable order records — and [Toward mainnet](./toward-mainnet.md) explains why that change, and the refill's write-ahead record, belong to the production phase rather than to a proof-of-concept.

## Explicitly out of scope for this proof-of-concept

The following are future work, called out so the boundary is unambiguous.

- **The real exchange purchase.** The automatic top-up is built and running: after a settlement's compressed holding period, Troia refills the pool from the lira that order collected, at the live exchange rate, so the pool grows by the commission. What is deferred is the single step that turns collected lira into genuinely acquired USDC on an exchange, rather than into test USDC minted by Troia. The treasury's cash-flow timing — refills follow the processor's settlement cadence, not how fast the pool drains — is covered on the [Architecture](./architecture.md) page.
- **Customer identity verification.** Unlike the boundaries below, this one does not exist in the code yet — there is no component, no interface, and no placeholder, and the hosted payment form carries a fixed buyer record. It arrives with the regulated production phase, which is where it belongs.
- **Hardware-backed keys and multi-signature thresholds.** The signing boundary already exists; on the test network a single signature is required, with the same flow shape as a stricter production setup.
- **Parallel payouts.** Today a single-writer counter processes payouts one at a time. Handling many at once is future work behind the same interface, and does not change any guarantee.
- **Third-party checkout extensions.** A bonus integration rail; the demonstration storefront does not depend on it.
- **A browser extension that works on any store.** The extension acts only on the storefront origins recorded in the deployment — Troia's own demo shop, never every site you visit. Widening that to a shop Troia does not control needs one thing first, and it is not built: the payment request on the page must be *signed* by the shop that published it. Today the extension checks that a request is well formed — a valid destination, an allowlisted issuer — but never who wrote it, so on an allowlisted origin an injected request could name any destination and still pass every check. That is not exploitable today, because the only allowlisted origin is Troia's own storefront: an injection there means the storefront itself was compromised, not a bystander site. It becomes real the moment a third-party shop joins the list — which is exactly what "works on any store" means. The fix keeps the design honest, and it needs no registry of merchants: verify the request's signature against the key the shop publishes at its own domain, and require that domain to be the one the page was actually served from.
- **Mainnet.** Going live is treated as a separate, regulated phase. No regulator has been contacted yet, and the sequencing is deliberate: prove the settlement and verification layers where a mistake costs nothing, then buy the licence that lets the system hold real money. Reversing that order means paying for compliance on a system nobody has yet shown to be correct.

## How to hold Troia to this

Everything claimed under *What is proven today* is checkable from a clone of the repository: run the test gate, and run the offline verification tool. Everything below it is stated as a limitation precisely so it cannot be mistaken for a claim. If any future page or demonstration asserts that something on this page is done, treat this page as the contradicting authority until it is updated with evidence.
